# Backend Architecture

Backend stack:

- PostgreSQL
- Supabase
- RPC / Stored Procedures for write operations
- Direct queries allowed for reads

General rule:

READS → queries  
WRITES → RPC / stored procedures

The application should avoid performing direct inserts/updates in
business-critical tables when an RPC exists.

---

# Core Business Flow

The ERP currently focuses on closing the commercial and budgeting
pipeline up to adjudication.

Main flow:

customers
→ orders
→ order_versions
→ proposals
→ adjudications

Future phases (project / production / after-sales) are not yet the
primary scope.

# Important RPC / Stored Procedures

The application should use RPC functions for most write operations.

Key functions include:

create_order(...)  
Creates a new order and initial version.

create_order_revision(...)  
Creates a new revision of an order.

create_order_revision_from_crm(...)  
Creates order revision directly from CRM flow.

assign_budgeter(...)  
Assigns budgeting team members to an order version.

update_order_budget_details(...)  
Updates budgeting metadata.

update_budget_assignment_hours(...)  
Updates worked hours for budgeting assignments.

send_proposal(...)  
Marks a proposal as sent and advances workflow.

set_order_commercial_phase(...)  
Updates the commercial workflow phase.

update_latest_order_version_expected_delivery(...)  
Updates expected delivery date of latest version.

# Important Business Constraints

order_ref

Generated automatically via trigger.

revision_code

Expected format: single uppercase letter.

assignment_role

Allowed values:

lead  
support

proposal validity

Updating valid_until automatically manages proposal follow-up.

# Workflow Phases

Workflow phases are controlled through workflow_phases table.

Examples

COM phases → commercial process  
ORC phases → budgeting process  
PRJ phases → project phase  
PRO phases → production phase

These phases drive the state machine of the order lifecycle.
